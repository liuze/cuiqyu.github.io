三、永久代溢出

  1. 永久代溢出-常量池溢出：

/**

  • 永久代常量池溢出
    */
    private static void constantPoolSpillover() {
    List list = new ArrayList();
    int i = 1;
    try {
    while (true) {
    list.add(UUID.randomUUID().toString().intern());
    i++;
}
    } finally {
    System.out.println("总共运行次数：" + i);
    }
    }

在jdk1.6及以前的版本中，由于常量池是分配在永久代内；
所以我们可以通过-XX:PermSize和-XX:MaxPermSize限制方法区大小；

从而间接限制其中常量池的容量；此时控制台打印：

Exception in thread "main" java.lang.OutOfMemoryError: PermGen space  
at java.lang.String.intern(Native Method)  
at com.limpid.MemoryTest.main(MemoryTest.java:18)

从运行结果中可以看到，运行时常量池溢出，在OutOfMemoryError后面跟随的提示信息是

“PermGen space”，说明运行时常量池属于方法区（HotSpot虚拟机中的永久代）的一部分

在jdk1.6之后的版本，如1.7中，就不会出现上面的情况，while循环会一直运行下去
——————————–总–结—————————————-
1、在jdk1.6及以前版本中，String.intern()方法会把首次遇到的字符串实例复制到永久
代中，返回的也是永久代中这个字符串实例的引用。
2、在jdk1.7及以后版本中（以及部分其他虚拟机，例如JRockit）String.intern()实现不
会再复制实例，只是在常量池中记录首次出现的实例引用

方法区用于存放Class的相关信息，如类名、访问修饰符、常量池、字段描述、方法描述等

四、直接内存溢出

总结

  1. 堆内存溢出：
    • 原因：不断的new一个对象，一直创建新的对象，直到jvm没有可以分配的堆内存
    • 解决：如果是内存泄露则看对象如何被GC Root引用，不足则通过调大-Xms，-Xmx参数
  2. 栈内存溢出：
    • 原因：死循环或者是递归太深，递归的原因，可能太大，也可能没有终止
      • 是否有递归调用
      • 是否有大量循环或死循环
      • 全局变量是否过多
      • 数组、List、map数据是否过大
      • 使用DDMS工具进行查找大概出现栈溢出的位置
    • 解决：
      • 是否App中的类中和引用变量过多使用了Static修饰 如public staitc Student s；在类中的属性中使用 static修饰的最好只用基本类型或字符串. 如public static int i = 0;
      • 是否App中使用了大量的递归或无限递归（递归中用到了大量的建新的对象）
      • 是否App中使用了大量循环或死循环（循环中用到了大量的新建的对象）
  3. 永久代溢出：
    • 原因：放入常量池的常量太多，导致常量池溢出；方法区溢出（类名、访问修饰符、常量池、字段描述、方法描述等）
    • 解决：
      • 字符串存在永久代中，容易出现性能问题和内存溢出
      • 类及方法的信息等比较难确定其大小，因此对于永久代的大小指定比较困难，太小容易出现永久代溢出，太大则容易导致老年代溢出
      • 永久代会为 GC 带来不必要的复杂度，并且回收效率偏低。
  4. 直接内存溢出：
    • 原因：直接内存（DirectMemory）可以通过-XX:MaxDirectMemorySize指定，如果不指定，默认与Java堆的最大值(-Xmx指定)一样，当超过该内存时就会导致直接内存溢出。
    • 解决：通过修改-XX:MaxDirectMemorySize配置来指定直接内存大小
